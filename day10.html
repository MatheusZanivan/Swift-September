UIKit essentials

Matheus Z. Andrade | iOS Engineer

Introduction

Whether you’re just starting with UIKit or refining a production app, mastering the essentials helps you write cleaner code, avoid frustrating layout bugs, and build experiences that adapt gracefully across devices. This article covers the fundamentals: view controller lifecycle, Auto Layout, safe areas, trait collections, and accessibility-friendly features like Dynamic Type. Let’s walk through the pieces that make UIKit apps robust and future-proof.

UIViewController lifecycle

The view controller lifecycle is the backbone of UIKit. Each method has a role: viewDidLoad for one-time setup, viewWillAppear for refreshing UI, and viewDidAppear for animations or analytics. Save state or stop work in viewWillDisappear and viewDidDisappear. Avoid expensive work in layout passes.

Copied
class HomeViewController: UIViewController {
  override func viewDidLoad() {
    super.viewDidLoad()
    // One-time setup
    view.backgroundColor = .systemBackground
  }
‎
  override func viewWillAppear(_ animated: Bool) {
    super.viewWillAppear(animated)
    // Refresh data or UI
  }
‎
  override func viewDidAppear(_ animated: Bool) {
    super.viewDidAppear(animated)
    // Start animations or analytics
  }
}

Auto Layout fundamentals

Prefer constraints over manual frames. Auto Layout respects device variations and system features. Use intrinsicContentSize, content hugging, and compression resistance to reduce ambiguity and avoid conflicts.

Copied
let label = UILabel()
                ‎
label.translatesAutoresizingMaskIntoConstraints = false
label.text = "Hello Auto Layout"
‎
view.addSubview(label)
‎
NSLayoutConstraint.activate([
  label.centerXAnchor.constraint(equalTo: view.centerXAnchor),
  label.centerYAnchor.constraint(equalTo: view.centerYAnchor)
])

Safe areas & layout guides

Safe areas prevent your UI from colliding with notches, toolbars, and the home indicator. Use safeAreaLayoutGuide for positioning and readableContentGuide for comfortable text layouts on larger devices.

Copied
let button = UIButton(type: .system)
                ‎
button.setTitle("Tap me", for: .normal)
button.translatesAutoresizingMaskIntoConstraints = false
‎
view.addSubview(button)
‎
NSLayoutConstraint.activate([
  button.bottomAnchor.constraint(equalTo: view.safeAreaLayoutGuide.bottomAnchor, constant: -20),
  button.centerXAnchor.constraint(equalTo: view.centerXAnchor)
])

Trait collections & adaptation

Size classes (.compact, .regular) and display scale help apps adapt to different environments. Override traitCollectionDidChange(_:) when you need to react dynamically, such as multitasking on iPad. Prefer declarative logic over device-specific checks.

Copied
override func traitCollectionDidChange(_ previousTraitCollection: UITraitCollection?) {
  super.traitCollectionDidChange(previousTraitCollection)
  ‎
  if traitCollection.horizontalSizeClass == .compact {
    // Adjust layout for compact width
  }
}

Dynamic Type & appearances

Support accessibility by opting into Dynamic Type and system colors. Setting adjustsFontForContentSizeCategory = true ensures your text scales. Use semantic colors like .label to automatically adapt to Dark Mode.

Copied
let label = UILabel()
                ‎
label.font = UIFont.preferredFont(forTextStyle: .body)
‎
label.adjustsFontForContentSizeCategory = true
‎
label.textColor = .label

Conclusion

UIKit remains a core framework for iOS development. By respecting the lifecycle, leaning on Auto Layout, honoring safe areas, and embracing adaptation APIs, your apps will look polished across devices. Accessibility and system features like Dark Mode are not extras—they are table stakes for modern, inclusive apps.

Next steps

Try adding a label and button to a new view controller, constrain them with Auto Layout, and test across iPhone and iPad simulators. Then enable Dynamic Type and Dark Mode to see how your UI reacts. Practice adapting with traitCollectionDidChange(_:) by simulating multitasking on iPad.
